01 个人知识管理的痛点

每个人的资料收藏都在不断增长——收藏的文章、下载的 PDF、会议笔记、技术文档。但当资料越来越多，我们往往会发现一个矛盾：

知识在膨胀，但管理能力没有跟上。

具体表现为五个方面：

  • 找不到——收藏的文章、笔记散落在各处，用的时候翻不到
  • 记不住——看过就忘，下次遇到同样的问题还要重新搜索
  • 信不过——不同来源说法矛盾，不知道信哪个
  • 理不动——整理笔记费时，越攒越多，最后放弃
  • 查得慢——查一个数据要打开十几个文件

核心命题

当个人收集的资料越来越多时，传统笔记方式无法应对知识膨胀。需要一种"用对话管理知识"的新范式。

02 什么是 LLM-Wiki？

LLM-Wiki 是一个基于大语言模型的个人知识库系统。概念源自 Andrej Karpathy（前 Tesla AI 总监、OpenAI 联合创始人）提出的 "LLM Wiki" 模式。

它的核心思想很简单：用 AI 自动读取、编写、维护结构化的 Markdown 知识库，而不是靠人手动整理。

你只需要通过四个 Slash Commands 就能完成所有操作：

  • /wiki-ingest raw/doc.pdf —— 导入文档，AI 自动提取、整理、索引
  • /wiki-query 你的问题 —— 自然语言提问，系统智能路由回答
  • /wiki-lint —— 健康检查，自动检测并修复问题
  • /wiki-graph —— 构建知识图谱，可视化知识关联

当然，也可以直接用自然语言描述需求，AI 会自动识别并执行对应工作流。

来源可追溯，不"幻觉"

LLM-Wiki 遵循来源优先原则——这是区别于普通 AI 对话的关键：

  • 资料中有 —— 引用具体页面来源（[[wikilink]]）
  • 资料中没有 —— 明确告知"资料中未提及"，不会编造
  • 原文档回溯 —— wiki 信息不足时，回到原始 PDF/PPTX/Excel 提取细节

Karpathy 模式核心

传统笔记是为人类浏览写的，LLM-Wiki 是为 AI 推理优化的。文档结构精简、语义明确、链接可解析，让 AI 能高效理解、推理和回答。每条回答都带引用，可随时追溯到原始来源。

03 核心能力与工作流

六大核心能力

  • 自然语言导入——丢入文档，AI 自动提取摘要、建立实体页和概念页、更新索引
  • 智能查询——用自然语言提问，系统自动选择最佳路径回答，每条回答带来源引用
  • 自我健康检查——自动检测断链、孤儿页面、矛盾内容，生成报告并自动修复
  • 知识图谱——自动提取语义关系，构建可视化图谱，发现隐藏的知识关联
  • 版本控制——每次操作自动提交 Git，完整变更历史，可随时回溯
  • AI 优化存储——文档结构为 AI 推理优化，相比 RAG 可减少约 95% 的 Token 消耗

工作流详解

导入（Ingest）

输入 /wiki-ingest raw/my-document.pdf，AI 自动完成：

  1. 读取源文档全文
  2. 提取关键信息，写入摘要页
  3. 创建/更新实体页（人、公司、项目、产品）
  4. 创建/更新概念页（技术、框架、方法）
  5. 自动更新全局综合摘要
  6. 检测与现有内容的矛盾
  7. 更新索引和日志，自动提交版本

查询（Query）

输入 /wiki-query 你的问题，系统智能路由：

查询类型路径Token 消耗
结构化数据（规格/参数/性能）数据库几乎为零
概念解释 / 技术分析Wiki低
跨文档对比 / 开放分析Agent中

健康检查（Lint）

自动检测：孤儿页面、断链、矛盾、过期摘要、缺失实体页、数据缺口，生成报告并自动修复。

知识图谱（Graph）

  • Pass 1：解析所有页面链接 —— 确定性关系边（零 Token）
  • Pass 2：AI 推断语义关系（依赖、矛盾、扩展等）
  • 社区检测自动聚类（Louvain 算法，零 Token）
  • 输出交互式 HTML 可视化

为什么 LLM-Wiki 比传统方案更好？

LLM-Wiki 概念由 Andrej Karpathy 提出并在社区引起广泛讨论。以下是与常见方案的对比：

与 Naive RAG（传统向量检索）对比

Naive RAG
  • 将文档切片为 vector chunks，靠余弦相似度检索
  • 容易检索到无关片段，丢失上下文
  • 每次查询都需要加载大量 tokens
  • 结构化数据（数字、表格）表现差
  • 需要维护向量数据库和 embedding 服务
LLM-Wiki（结构化知识）
  • AI 主动构建结构化知识，而非被动检索
  • 每个知识点有明确的位置和引用链
  • 渐进式加载，减少约 95% Token 消耗
  • SQLite 处理结构化数据，精确返回数字
  • 纯 Markdown 文件 + Git，零运维

与传统笔记工具（Obsidian / Notion）对比

维度Obsidian / NotionLLM-Wiki
知识构建人工编写为主AI 自动提取+整理
文档导入手动粘贴/导入自动读取 PDF/PPTX/Excel
知识维护手动整理断链Lint 自动检测+修复
知识查询关键词搜索自然语言问答
知识关联手动加链接自动提取+图谱构建
矛盾检测无自动检测+报告
结构化数据表格手动维护SQLite 自动查询

全方位对比总结

维度传统笔记RAG / 向量库LLM-Wiki
知识录入手动自动切片AI 自动提取+整理
知识查询关键词搜索向量检索自然语言+智能路由
精确性高（人工确认）中（相似度匹配）高（来源引用）
Token 效率N/A低（全量加载）高（渐进式，-95%）
自我维护无无Lint 自动修复
版本控制内置历史无Git 完整版本
部署成本订阅费向量库+服务Git 仓库（免费）
为 AI 优化否部分是

LLM-Wiki 不是要替代 RAG 或笔记工具，而是补足了它们的短板——RAG 的精确性不足、传统笔记的维护成本过高。LLM-Wiki 用"文件系统即数据库"的极简哲学，把 Wiki 的精确性和 AI 的理解力结合起来。

05 系统架构

整个系统由两层组成：

源文档层（raw/）

不可变的原始文件——PDF、PPTX、Excel、网页等。只读不写，保证数据安全性。

知识层（wiki/）

AI 从源文档中提取、整理、生成的结构化知识，包括摘要页、实体页、概念页、全局索引和综合摘要。

数据流

源文档 → AI 读取分析 → 知识层构建 → 索引+图谱 → 版本控制

配合 SQLite 数据库处理结构化数据查询（信号、参数、性能指标），配合 渐进式加载避免上下文膨胀——只读与查询相关的页面，减少约 80% 无关内容。

目录结构

my-wiki/ ├── raw/ # 不可变源文档（PDF/PPTX/Excel/MD） ├── wiki/ # AI 生成的知识层 │ ├── index.md # 全页面索引 │ ├── overview.md # 全局综合摘要 │ ├── log.md # 操作日志（追加式） │ ├── sources/ # 每份源文档的摘要页 │ ├── entities/ # 人、公司、项目、产品 │ ├── concepts/ # 概念、框架、方法 │ └── syntheses/ # 保存的查询分析结果 ├── graph/ # 知识图谱（graph.json + graph.html） ├── tools/ # 本地工具脚本（db_query.py, build_graph.py） ├── data/ # SQLite 数据库文件 ├── .claude/ │ └── commands/ # Slash Commands（wiki-ingest/query/lint/graph） └── CLAUDE.md # 工作流配置（AI 的"大脑指令"）

本项目已包含以下文件，clone 后开箱即用：

  • .claude/commands/wiki-ingest.md —— /wiki-ingest 导入文档
  • .claude/commands/wiki-query.md —— /wiki-query 自然语言提问
  • .claude/commands/wiki-lint.md —— /wiki-lint 健康检查
  • .claude/commands/wiki-graph.md —— /wiki-graph 构建知识图谱

已安装 Claude Code（npm i -g @anthropic-ai/claude-code）并配置好 API Key。创建 raw/ 和 wiki/ 目录后即可开始使用导入和查询功能。数据库和图谱构建是可选增强。

07 与 Obsidian 集成

LLM-Wiki 的 wiki/ 目录本身就是标准 Markdown 文件，可以直接作为 Obsidian 的 Vault 使用。集成后你可以获得：

  • 双向链接可视化——Obsidian 原生支持 [[wikilink]] 语法，自动展示知识关联图
  • 实时预览编辑——在 Obsidian 中直接浏览和编辑 wiki 页面
  • 标签系统——利用 frontmatter 中的 tags 字段进行分类筛选
  • 本地离线访问——所有数据在本地，无需联网即可查看知识库

基本配置步骤

1
打开 Obsidian Vault
在 Obsidian 中选择 "Open folder as vault"，将项目的 wiki/ 目录（或项目根目录）设为 Obsidian 的 Vault。这样 Obsidian 就能识别所有 Markdown 文件和 [[wikilink]] 链接。
2
配置 Obsidian 别名（可选）
如果你希望用中文别名访问英文页面，可以在 wiki 页面的 frontmatter 中添加 aliases 字段。例如在 MT8678.md 中添加 aliases: [MTK8678]，即可通过 [[MTK8678]] 访问同一页面。
3
启用 Graph View
在 Obsidian 左侧面板点击 "Graph view" 图标，即可看到基于 [[wikilink]] 构建的知识图谱。这与 LLM-Wiki 自动生成的 graph/graph.html 互补——Obsidian 展示的是静态链接关系，graph.html 包含 AI 推断的语义关系。
4
Git 插件同步（推荐）
安装 Obsidian 的 obsidian-git 插件，设置自动提交。这样在 Obsidian 中手动编辑的页面也会通过 Git 同步，与 Claude Code 自动生成的内容保持一致。

目录对应关系

Obsidian Vault 根目录 → 项目根目录（或 wiki/ 子目录）。两种方式均可：如果 Vault 指向项目根目录，Obsidian 可以看到 raw/ 和 wiki/ 的完整结构；如果只指向 wiki/ 子目录，则专注于知识层的浏览和编辑。

08 成本说明（API Key + Tokens）

必要前提

LLM-Wiki 需要大语言模型的 API 支持，必须购买并配置 API Key。推荐使用 Claude（Anthropic）系列模型，也支持其他兼容 API 的 LLM。所有 AI 操作（导入、查询、Lint、图谱构建）都会消耗 Tokens。

Token 消耗场景

操作Token 消耗级别说明
导入文档（Ingest）高需要读取完整文档 + 提取摘要 + 创建多个页面
查询（Query）低 ~ 中渐进式加载只读相关页面；DB 查询几乎不消耗 Tokens
知识图谱构建高需要分析所有页面的语义关系
健康检查（Lint）低主要靠脚本检查，AI 只参与矛盾检测

成本优化策略

  • 渐进式加载——三级索引体系，只加载与查询相关的页面，减少约 80% 无关内容
  • 结构化查询优先——数字、规格等结构化数据走 SQLite DB，几乎零 Token 消耗
  • 增量图谱构建——使用 SHA256 缓存，只处理变更的页面，避免全量重建
  • 模型分级使用——Ingest 用大模型（高质量提取），Query 用中小模型

LLM-Wiki 需要 API Key 和 Token 费用，但由于渐进式加载和结构化查询优先的设计，实际 Token 消耗远低于 Naive RAG。加上零基础设施成本（无需服务器、无需向量库），总体拥有成本更低。
