## 一句话摘要

该材料描述了一套通过日志提取 GPS / GNSS 数据、计算惯导相关 cn0 信号强度、生成 KML 轨迹并导入地图的方法，用于分析车辆进入隧道后 GPS 或惯导输出是否存在偏移、漂移或不符合预期的问题。

## 背景

材料中的场景来自某项目 bug：客户反馈在某时间点 GPS 定位不准、漂移或不动。由于 GPS 数据无法直接高效分析，且一段时间内存在多条 GPS 定位点，逐点在地图上确认效率较低，因此需要从日志中提取 GPS 数据并转换为打点信息，在地图中绘制实际行车轨迹。

材料特别说明：文档示例基于反馈的惯导数据打点偏移；对于模组输出的卫星导航数据，整体分析流程大致一致。

## 目标

以大通项目为例，提供一个完整的分析打点数据流程：

1. 根据 bug 提供的时间点和日志，快速提取 GPS 数据文件。
2. 在地图中生成路径轨迹信息。
3. 判断客户反馈的定位偏移是否超过 CP 定位标准要求。
4. 辅助区分问题可能来自模组输出、惯导输出或上层导航应用绘制。

## 主要分析步骤

### 1. 确认进入隧道或发生问题的大致时间

- 在 Android 日志中搜索 `nmea data report GGA`，提取对应 GPS 数据。
- 材料中的分析逻辑是：车辆进入隧道后卫导数据失效，对于日志来说经纬度信息保持不变。
- 通过开始时间戳和结束时间戳筛选问题时间段内的 GPS 数据。

### 2. 分析 cn0

- 根据大致时间点找到对应惯导日志，例如 `dr_logxxxx`。
- 使用供应商提供的分析工具导入数据，计算 cn0。
- 图中 cn0 急速衰减到 0，材料判断这代表对应时间确实收到数据干扰，也就是进入了隧道。
- 材料认为分析 cn0 不是打点流程的必要步骤，但可以用于判断进入隧道或普通时间段 GPS 信号强度。
- 对于正常 GPS 数据，cn0 最小不应低于 40；该阈值来自材料描述，需后续确认适用范围和来源。

### 3. 提取打点数据并生成 KML 文件

- 根据问题时间戳，在惯导输出数据中找到对应时间间隔。
- 在日志中找到 `dr_nmea` 开头的对应日志。
- 使用脚本 `gnrmc_kml_by_time.py` 提取打点数据并生成 KML 文件。
- 示例命令：

```bash
python gnrmc_kml_by_time.py dr_nmea_20250731110155_1.txt
```

材料中还给出了带开始/结束时间戳的命令格式：

```bash
gnrmc_kml_by_time.py dr_nmea_xxx.txt <开始时间戳> <结束时间戳>
```

### 4. 导入地图查看轨迹

- 打开 Google My Maps：`https://www.google.com/maps/d/`
- 将生成的 KML 打点数据导入地图，得到轨迹图。
- 材料示例中展示了：
  - 约 2 km 偏移。
  - 计算出的隧道偏移。

## 材料结论

材料认为：

1. 该方法可以在客户强调 GPS 数据输出偏移等问题时，给出清晰直观的分析。
2. 它可以辅助判断问题来自自模组或惯导数据输出。
3. 它也可以用于区分“模组打点输出准确，但上层导航 app 定位点绘制出错”的情况。

## 适用场景

- GPS / GNSS 定位不准、漂移、不动。
- 车辆进入隧道、地下或遮挡环境后定位异常。
- 需要验证模组输出、惯导输出或上层导航绘制之间责任边界。
- 需要把日志中的 NMEA / GGA / GNRMC 数据转换成地图轨迹。
