## 用户目标观察

- 用户是 Android Framework 技术 Leader。
- 用户希望用 Wiki 搭建个人/团队专属知识库，目标是沉淀技术资产、提升个人能力、赋能团队协作。
- 用户明确偏好“加入的要精，不要追求大”，即知识库应重质量、可用性和精选范围，而不是追求覆盖面。
- 第一阶段服务对象选择为“个人为主，团队可读”：用户主导沉淀，团队先作为读者使用，暂不强制共建。
- 用户希望知识类型覆盖 A/B/C/D：问题定位方法论、Framework 架构地图、项目实战复盘、团队技术标准。但仍保留“精选、不求大”的原则，因此需要通过分层和准入标准避免范围失控。
- 精选准入标准选择为“高频 / 高损 / 可复用，满足至少两个才入库”。
- 第一阶段初始规模选择为“按场景组织”：先选 5～8 个最高价值 Android Framework 场景，每个场景下沉淀相关的问题定位、架构、复盘和标准，而不是按篇数凑规模。
- 首批场景优先级选择为“混合首批场景”：从稳定性、Framework 核心链路、车机/多端复杂场景中各选高价值场景。
- 用户补充：知识库主要还会包含组员写的文档。这意味着知识库不仅是个人沉淀，也需要处理团队输入文档的筛选、归一化、质量门槛和引用溯源。
- 组员文档形态选择为“都有，但质量参差不齐”：包括问题分析、需求/方案设计、模块学习/源码阅读，以及过程记录、会议纪要或零散笔记。
- 组员文档进入 Wiki 的处理方式选择为 C + D：采用双层结构，原文进入 `raw/` 或 `sources/`，精选内容进入 `wiki/`；同时先进入 inbox 临时收集，再定期筛选进入正式 Wiki。
- 正式 `wiki/` 精选页模板选择为 D：按内容类型使用不同模板；问题分析用问题定位模板，架构地图用架构链路模板，项目复盘用决策模板，团队标准用 checklist 模板；所有页面统一要求包含“结论 / 适用场景 / 来源”。
- 组员文档提交入口选择为 D：新增文档尽量按轻量模板提交到 inbox；历史文档由用户定期捞取和精选。
- Wiki 维护节奏选择为 D：每周轻量 triage 处理 inbox，双周或月度围绕专题做收敛和重构。
- 第一阶段成功标准选择为 D：个人有效、团队开始复用、文档质量变好；重点看团队是否真的开始引用 Wiki。建议量化目标：4 周内至少 5 次团队讨论、问题分析或方案评审中引用 Wiki 页面。
- 总体设计方向确认采用“方案三：双轴精选 Wiki”：以场景作为主入口，以资产类型作为页面模板和维护维度。
- 用户确认信息架构方向可行。
- 用户确认四类精选资产模板方向可行：问题定位模板、架构链路模板、项目/问题复盘模板、团队标准/Checklist 模板；第一阶段强制字段只保留“一句话结论 / 适用场景 / 来源”。
- 用户确认文档流转流程方向可行：`inbox -> triage -> source -> asset`，通过每周轻量筛选和双周/月度专题收敛，避免正式 Wiki 被低质量输入污染。
- 用户确认维护机制方向可行：第一阶段由组员提交原始输入，用户负责筛选和归一化，团队阅读、引用、反馈；每周 triage，双周/月度专题收敛；4 周内以团队引用和文档质量提升作为成功指标。
- 用户补充：Wiki 在一定程度上也需要承担资料库能力。组员写的使用/操作文档、三方给的重要资料，也可以入知识库，目标是方便以后查询。这要求设计从“纯精选知识资产库”调整为“精选知识资产层 + 受控资料层”的双层模型。
